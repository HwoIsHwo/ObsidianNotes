Хеш-таблица — это структура данных для хранения пар «ключ–значение» с ожидаемой сложностью операций O(1) на вставку, поиск и удаление. Основная идея проста: вместо линейного поиска по всем элементам ключ пропускается через хеш-функцию, которая преобразует его в число — индекс внутри внутреннего массива.

Базовая модель выглядит так:
key ──> hash(key) ──> index ──> bucket\[index]

Внутри таблицы есть массив бакетов фиксированного размера. Хеш-функция вычисляет значение типа `size_t`, затем берётся остаток от деления на количество бакетов. Это определяет, в какой бакет попадёт элемент.
Однако хеш-функция отображает большое пространство ключей в конечное число индексов. Поэтому коллизии неизбежны — разные ключи могут попасть в один и тот же бакет. Вопрос не в том, бывают ли коллизии, а в том, как их обрабатывать.


Существует два основных подхода к разрешению коллизий.
Первый — цепочки (separate chaining). Каждый бакет хранит список элементов. Если несколько ключей дали одинаковый индекс, они просто добавляются в список внутри этого бакета. Поиск в среднем остаётся O(1), но в худшем случае может деградировать до O(n), если все элементы попадут в один бакет. Именно такой подход используется в большинстве реализаций `std::unordered_map`.
bucket\[3]:
    (key1, value1) -> (key2, value2) -> (key3, value3)

Второй подход — открытая адресация (open addressing). В этом случае элементы хранятся прямо в массиве, а при коллизии выполняется пробирование: поиск следующей свободной позиции по определённому правилу (линейное, квадратичное, двойное хеширование). Здесь нет отдельных списков, но важно следить за коэффициентом заполнения.


Коэффициент загрузки (load factor) — это отношение числа элементов к числу бакетов. Когда он становится слишком большим, таблица расширяется (rehash): создаётся новый массив большего размера, и все элементы перераспределяются. Это дорогая операция, но она происходит редко, поэтому средняя сложность остаётся амортизированно O(1).
Производительность хеш-таблицы определяется тремя факторами:
- качеством хеш-функции
- способом разрешения коллизий
- коэффициентом загрузки
Плохая хеш-функция может привести к тому, что большинство элементов окажутся в одном бакете. Тогда структура фактически превращается в связный список, и сложность операций станет линейной.

В C++ хеш-таблица представлена контейнерами `std::unordered_map` и `std::unordered_set`. Они используют хеш-функцию `std::hash` и оператор равенства для проверки совпадения ключей. Важно соблюдать контракт: если `a == b`, то их хеши обязаны совпадать. Обратное не требуется.

```
#include <unordered_map>

std::unordered_map<std::string, int> table;

table["one"] = 1;
table["two"] = 2;

auto it = table.find("one");
if (it != table.end()) {
    std::cout << it->second;
}
```

Здесь поиск не требует перебора всех элементов — выполняется вычисление хеша и переход к нужному бакету.
По памяти хеш-таблица обычно менее компактна, чем `std::map`. Она хранит массив бакетов плюс узлы (в случае цепочек), что даёт дополнительный overhead. Зато она быстрее при случайном доступе.
Важно понимать, что хеш-таблица не сохраняет порядок элементов. Итерация происходит в порядке, зависящем от распределения по бакетам. Если нужен упорядоченный контейнер, используется дерево (например, `std::map`), которое обеспечивает логарифмическую сложность и сортировку по ключу.
С точки зрения архитектуры хеш-таблица — это компромисс между временем доступа и дополнительной памятью. Она эффективна, когда нужен быстрый доступ по ключу и порядок элементов не важен. Если же важны диапазонные запросы, сортировка или минимальное потребление памяти, более подходящими могут быть другие структуры.
В практическом коде хеш-таблицы особенно полезны для кэширования, индексации, частотных таблиц, отображения идентификаторов в объекты и любых сценариев, где доминируют операции поиска по ключу.
