**KISS (Keep It Simple, Stupid)** — принцип: решение должно быть максимально простым из работающих. 

Он направлен против:
- преждевременной абстракции
- усложнённых иерархий
- универсальности «на всякий случай»
- лишней обобщённости

## Что значит «простое» решение
Простота — это:
- минимальное количество сущностей
- очевидная логика
- отсутствие ненужных уровней абстракции
- читаемость без объяснений
Важно: простота ≠ меньше строк кода.  
Иногда более длинный, но прямой код проще для понимания.

Пример нарушения KISS
```
template<typename T>
class IDataProcessorStrategy {
public:
    virtual ~IDataProcessorStrategy() = default;
    virtual T process(const T&) = 0;
};
```
Если в проекте всего одна реализация и не планируется расширение — это переинжиниринг.
Проще:
```
int process(int value) {
    return value * 2;
}
```


## Пример с классом
Плохо:
```
class UserManagerFactory;
class IUserManager;
class UserManagerImpl;
```
Если система небольшая и не предполагается подмена реализации — фабрика и интерфейс могут быть лишними.
Просто:
```
class UserManager {
public:
    void add_user(const std::string& name) {
        users_.push_back(name);
    }
private:
    std::vector<std::string> users_;
};
```

## Где чаще всего нарушают KISS
1. Преждевременный полиморфизм Создают интерфейс «на будущее», которого может не быть.
2. Чрезмерные шаблоны Обобщение там, где нужен конкретный тип.
3. Избыточные паттерны Factory, Strategy, Visitor — когда достаточно обычной функции.

## KISS vs SOLID
Иногда между ними возникает напряжение.
SOLID может приводить к увеличению числа классов.  
KISS говорит: не усложняй без реальной причины.
Правильный баланс:
- сначала простое решение
- усложнять только при появлении реальной необходимости
