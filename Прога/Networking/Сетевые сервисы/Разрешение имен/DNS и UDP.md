DNS — отличный пример сервиса прикладного уровня, использующего UDP в качестве транспортного уровня вместо TCP. Важно помнить, что главное отличие между TCP и UDP заключается в том, что UDP является протоколом без установления соединения, то есть нет необходимости устанавливать или разрывать соединение. Таким образом, общий объем передаваемого трафика значительно уменьшается. Один DNS-запрос и ответ обычно помещаются в одну UDP-дейтаграмму, что делает его идеальным кандидатом для протокола без установления соединения. Стоит также отметить, что DNS может генерировать большой объем трафика.

Давайте посмотрим, как будет выглядеть полный DNS-запрос через TCP. Сначала хост, отправляющий запрос на разрешение DNS, отправляет SYN-пакет локальному серверу имен на порт 53, который является портом, на котором прослушивает DNS. Затем этот сервер имен должен ответить SYN-ACK-пакетом. Это означает, что исходный хост должен ответить ACK для завершения трехстороннего рукопожатия, то есть трех пакетов. Теперь, когда соединение установлено, исходный хост должен отправить сам запрос. Мне нужен IP-адрес для food.com, пожалуйста. Когда сервер имен получит этот запрос, он должен будет ответить еще одним подтверждением (ACK). Я получил ваш запрос для food.com,
у нас уже отправлено пять пакетов.

В нашем сценарии первый кэширующий сервер имен ничего не кэширует для food.com. Поэтому ему нужно связаться с корневым сервером имен, чтобы узнать, кто отвечает за домен верхнего уровня .com. Для этого потребуется трехстороннее рукопожатие: сам запрос, подтверждение (ACK) или запрос, ответ и затем подтверждение ответа. Соединение должно быть закрыто с помощью четырехстороннего рукопожатия. Это еще 11 пакетов, или всего 16.

Теперь, когда рекурсивный сервер имен получил правильный сервер имен для домена верхнего уровня, ему нужно повторить весь этот процесс, чтобы обнаружить надлежащий уполномоченный сервер имен. Это еще 11 пакетов, итого 27.

Наконец, рекурсивному серверу имен придется повторить весь процесс еще раз, взаимодействуя с авторитетным сервером имен, чтобы фактически получить IP-адрес food.com. Это еще 11 пакетов, итого 38.

Теперь, когда локальный сервер имен наконец получил IP-адрес food.com, он может наконец ответить на первоначальный запрос. Он отвечает DNS-резолверу, который изначально отправил запрос, а затем этот компьютер отправляет ACK, чтобы подтвердить получение ответа. Это еще два пакета, итого 40.

Наконец, TCP-соединение необходимо закрыть с помощью четырехстороннего рукопожатия. Это приводит нас к общему количеству как минимум 44 пакетов, необходимых для выполнения полностью рекурсивного DNS-запроса по TCP.

Теперь давайте посмотрим, как это будет выглядеть с UDP, спойлер: пакетов потребуется меньше. Исходный компьютер отправляет UDP-пакет своему локальному серверу имен на порт 53, запрашивая IP-адрес food.com — это один пакет.

Локальный сервер имен действует как рекурсивный сервер и отправляет UDP-пакет корневому серверу, который отправляет ответ, содержащий правильный сервер имен верхнего уровня (TLD) — это три пакета.

Рекурсивный сервер имен отправляет пакет серверу TLD и получает ответ, содержащий правильный сервер авторизации — теперь у нас пять пакетов.

Далее рекурсивный сервер имен отправляет свой последний запрос серверу имен, который отправляет ответ, содержащий IP-адрес food.com — это семь пакетов. Наконец, локальный сервер имен отвечает DNS-резолверу, который изначально сделал запрос, IP-адресом food.com.

В итоге у нас получается восемь пакетов. Это идеальный пример того, почему существуют такие протоколы, как UDP, в дополнение к более надёжному TCP. Возможно, вы задаетесь вопросом, как здесь работает восстановление после ошибок, поскольку в UDP его нет. Ответ довольно прост. DNS-сервер просто запрашивает подтверждение ещё раз, если не получает ответа. По сути, DNS обеспечивает ту же функциональность, что и TCP на транспортном уровне, на прикладном уровне, в самом простом виде.

DNS-серверу не нужно ничего делать, кроме как отвечать на входящие запросы, а DNS-серверу достаточно просто выполнять запросы и повторять их, если они не удаются. Настоящая демонстрация простоты как DNS, так и UDP. Следует отметить, что DNS поверх TCP действительно существует и используется повсеместно.
