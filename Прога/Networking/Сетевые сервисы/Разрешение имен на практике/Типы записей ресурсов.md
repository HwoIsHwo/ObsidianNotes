DNS на практике работает с набором определенных типов записей ресурсов. Они позволяют осуществлять различные виды разрешений DNS. Существуют десятки различных типов записей ресурсов, но многие из них служат только для очень специальных целей.

Наиболее распространенная запись ресурса известна как <u>запись A</u>. Запись A используется для указания определенного доменного имени на определенный IP-адрес IPv4. В самом простом случае одна запись A настраивается для одного доменного имени, но одно доменное имя может иметь и несколько записей A. Это позволяет использовать технику, известную как <u>DNS round robin</u>, для балансировки трафика между несколькими IP-адресами.

<u>Round robin</u> — это концепция, которая предполагает последовательное прохождение списка элементов один за другим в определенном порядке. Цель состоит в том, чтобы обеспечить достаточно равномерное распределение нагрузки между всеми элементами выбранного списка. Допустим, мы отвечаем за доменное имя \www.microsoft.com. Чтобы сбалансировать трафик между несколькими серверами, мы настраиваем четыре записи A для \www.microsoft.com на авторитетном сервере имен для домена microsoft.com. Мы будем использовать IP-адреса 10.1.1.1, 10.1.1.2, 10.1.1.3 и 10.1.1.4. Когда DNS-резолвер выполняет поиск \www.microsoft.com, все четыре IP-адреса будут возвращены в порядке, в котором они были сначала настроены. 10.1.1.1, затем 10.1.1.2, затем 10.1.1.3 и, наконец, 10.1.1.4. Компьютер, выполняющий разрешение DNS, будет знать, что ему следует попробовать использовать первую запись, 10.1.1.1. Но он знает обо всех четырех на случай, если соединение с 10.1.1.1 не удастся. Следующий компьютер, выполняющий поиск \www.microsoft.com, также получит все четыре IP-адреса в ответе, но порядок изменится. Первой записью будет 10.1.1.2, за ней последует 10.1.1.3, затем 10.1.1.4, и, наконец, 10.1.1.1 будет последней в этом списке. Эта схема будет повторяться при каждой попытке разрешения DNS, циклически проходя по всем настроенным записям A и распределяя трафик между этими IP-адресами. Это основы работы логики DNS round robin.

Ещё один тип записей ресурсов, который становится всё более популярным, — это запись quad A. Запись quad A очень похожа на запись A, за исключением того, что она возвращает IPv6-адрес вместо IPv4-адреса.

Запись CNAME также очень распространена. Запись CNAME используется для перенаправления трафика с одного домена на другой. Допустим, Microsoft размещает свои веб-серверы по адресу \www.microsoft.com. Они также хотят убедиться, что любой, кто введёт в свой веб-браузер только microsoft.com, будет правильно перенаправлен. Настроив запись CNAME для microsoft.com, которая разрешается в \www.microsoft.com, клиент, осуществляющий разрешение, будет знать, что нужно выполнить ещё одну попытку разрешения, на этот раз для \www.microsoft.com, и затем использовать IP-адрес, полученный в результате этой второй попытки. Записи CNAME очень полезны, потому что они гарантируют, что вам нужно изменить канонический IP-адрес сервера только в одном месте. Фактически, CNAME — это просто сокращение от канонического имени. Если мы снова обратимся к нашему первоначальному примеру, когда нужно было убедиться, что посетители сайтов microsoft.com и www.microsoft.com попадают на один и тот же ресурс, то это можно сделать двумя способами. Мы могли бы создать идентичные A-записи для обоих доменов microsoft.com и www.microsoft.com. И это прекрасно работало бы. Но если базовый IP-адрес когда-либо изменится, нам нужно будет изменить его в двух местах: в A-записях для microsoft.com и www.microsoft.com. Создав CNAME, указывающий microsoft.com на www.microsoft.com, вам нужно будет изменить только A-запись для www.microsoft.com. И вы будете уверены, что клиенты, указывающие на любой из этих доменов, получат новый IP-адрес.

Ещё один важный тип записи ресурса — это запись MX. MX расшифровывается как mail exchange (обмен почтой), и эта запись ресурса используется для доставки электронной почты на нужный сервер. Многие компании запускают свои веб- и почтовые серверы на разных машинах с разными IP-адресами. Таким образом, запись MX позволяет легко гарантировать доставку электронной почты на почтовый сервер компании, в то время как другой трафик, например веб-трафик, будет доставляться на веб-сервер.

Тип записи, очень похожий на запись MX, — это запись SRV. SRV расшифровывается как service record (запись службы) и используется для определения местоположения различных конкретных служб. Она служит той же цели, что и тип записи ресурса MX, за исключением одного момента: в то время как MX предназначен только для почтовых служб, запись SRV может быть определена для возврата информации о многих различных типах служб. Например, записи SRV часто используются для возврата записей таких служб, как Cal Dave, которая представляет собой календарь и службу планирования.

Тип записи text — это интересный вариант. TXT расшифровывается как текст и изначально предназначался только для связывания описательного текста с доменным именем для восприятия человеком. Идея заключалась в том, чтобы можно было оставлять заметки или сообщения, которые люди могли бы обнаружить и прочитать, чтобы узнать больше о произвольных особенностях вашей сети. Поскольку текстовая запись имеет поле полностью свободной формы, находчивые инженеры придумали способы использовать ее для передачи данных, которые изначально не предназначались для передачи такими системами, как DNS. Эта текстовая запись часто используется для передачи параметров конфигурации сетевых служб, которые вы доверили другим организациям для обработки вашего домена. Например, часто текстовая запись используется для передачи дополнительной информации поставщику услуг электронной почты, то есть компании, которая занимается доставкой вашей электронной почты.
