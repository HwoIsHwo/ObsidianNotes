Подобно тому, как кадр Ethernet инкапсулирует IP-дейтаграмму, IP-дейтаграмма инкапсулирует сегмент TCP. Помните, что кадр Ethernet имеет раздел полезной нагрузки, который, по сути, представляет собой все содержимое IP-дейтаграммы. Также помните, что IP-дейтаграмма имеет раздел полезной нагрузки, который состоит из того, что известно как сегмент TCP.

Сегмент TCP состоит из заголовка TCP и раздела данных. Сам заголовок TCP состоит из множества полей, содержащих много информации.

Во-первых, у нас есть поля «порт источника» и «порт назначения». Порт назначения — это порт службы, для которой предназначен трафик. Порт источника — это порт с высоким номером, выбранный из специального раздела портов, известных как временные порты. Порт источника необходим для разделения множества исходящих соединений. Вы знаете, как порт назначения, скажем, порт 80, необходим для того, чтобы трафик достигал веб-сервера, работающего на определенном IP-адресе? Аналогично, порт источника необходим для того, чтобы, когда веб-сервер отвечает, компьютер, отправивший первоначальный запрос, мог отправить эти данные программе, которая фактически их запрашивала.

Далее идет поле, известное как «порядковый номер». Это 32-битное число, используемое для отслеживания того, где в последовательности TCP-сегментов ожидается этот сегмент.

Возможно, вы помните, что на более низком уровне стека протоколов существуют ограничения на общий размер передаваемых данных. Размер Ethernet-кадра обычно ограничен 1518 байтами, но нам обычно нужно передавать гораздо больше данных. На транспортном уровне TCP разбивает все эти данные на множество сегментов. Порядковый номер в заголовке используется для отслеживания того, к какому из множества сегментов может относиться данный конкретный сегмент.

Следующее поле, номер подтверждения, очень похоже на порядковый номер. Номер подтверждения — это номер следующего ожидаемого сегмента. Проще говоря, порядковый номер один и номер подтверждения два можно прочитать как «это сегмент 1, ожидайте следующий сегмент 2».

Далее следует поле смещения данных. Это поле представляет собой четырехбитное число, указывающее длину заголовка TCP для данного сегмента. Это необходимо для того, чтобы принимающее сетевое устройство понимало, где начинается фактическая полезная нагрузка данных.

Затем у нас есть шесть бит, зарезервированных для шести флагов управления TCP.

Следующее поле — это 16-битное число, известное как «окно TCP». Окно TCP определяет диапазон порядковых номеров, которые могут быть отправлены до того, как потребуется подтверждение. TCP — это протокол, который очень сильно зависит от подтверждений. Это делается для того, чтобы убедиться, что все ожидаемые данные действительно получены, и чтобы отправляющее устройство не тратило время на отправку данных, которые не были получены.

Следующее поле — это 16-битная контрольная сумма. Она работает так же, как поля контрольных сумм на уровне IP и Ethernet. После того, как весь сегмент принят получателем, вычисляется контрольная сумма по всему сегменту, которая сравнивается с контрольной суммой в заголовке, чтобы убедиться в отсутствии потери или повреждения данных в процессе передачи.

Поле «указатель срочности» используется совместно с одним из управляющих флагов TCP для указания конкретных сегментов, которые могут быть более важными, чем другие. Эта функция TCP практически не получила широкого распространения, и вы, вероятно, никогда не найдете её в современных сетях.

Далее у нас есть поле «опции». Как и поле «указатель срочности», оно редко используется в реальных условиях. Но иногда оно применяется в более сложных протоколах управления потоком.

Наконец, у нас есть заполнение, представляющее собой последовательность нулей, гарантирующую, что раздел полезной нагрузки данных начинается в ожидаемом месте.
