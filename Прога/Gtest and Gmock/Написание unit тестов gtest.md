Написание unit-тестов в gtest сводится к формализации ожидаемого поведения кода в виде проверок. Сам фреймворк не “знает”, что должна делать функция — это задаётся явно через assertions. Тест — это обычная функция без возвращаемого значения, внутри которой вызывается тестируемый код и проверяется результат.

Базовый сценарий всегда одинаковый: есть функция или класс, есть набор входных данных и есть ожидаемый результат. Это и есть содержание теста.

Рассмотрим минимальный пример.
```cpp
#include <gtest/gtest.h>

// тестируемая функция
int add(int a, int b) {
    return a + b;
}

// unit-тест
TEST(AddTest, HandlesPositiveNumbers) {
    EXPECT_EQ(add(2, 3), 5);  // проверяем результат
}
```

Макрос `TEST` принимает два аргумента: имя группы (TestSuite) и имя конкретного теста. Внутри — произвольный C++ код плюс assertions. Если хотя бы одна проверка не проходит, тест считается проваленным.
Ключевой момент — тест должен быть детерминированным. Никаких случайных значений, внешних зависимостей и побочных эффектов.

Точка входа стандартная:
```cpp
int main(int argc, char** argv) {
    ::testing::InitGoogleTest(&argc, argv);
    return RUN_ALL_TESTS(); // запускает все тесты
}
```
`RUN_ALL_TESTS()` возвращает код завершения, поэтому его обязательно возвращать из `main`.

Далее — assertions. Это основа всех тестов. Они проверяют условия и формируют результат теста.
```cpp
EXPECT_EQ(a, b);   // ожидаем равенство
EXPECT_NE(a, b);   // неравенство
EXPECT_TRUE(cond); // булево условие
```
Есть две категории:
- `EXPECT_*` — ошибка фиксируется, но тест продолжается
- `ASSERT_*` — тест немедленно прерывается
```cpp
ASSERT_NE(ptr, nullptr); // если nullptr — дальше идти нельзя
EXPECT_EQ(ptr->value, 10);
```
Практически это означает: `ASSERT_*` используется для проверок предусловий, `EXPECT_*` — для проверки результата.

Следующий уровень — покрытие разных сценариев. Один тест = один сценарий. Это важное правило, иначе при падении теста сложно понять, что именно сломалось.
```cpp
TEST(AddTest, HandlesZero) {
    EXPECT_EQ(add(0, 5), 5);
}

TEST(AddTest, HandlesNegative) {
    EXPECT_EQ(add(-2, -3), -5);
}
```
Такая декомпозиция делает тесты читаемыми и диагностируемыми.

Отдельный важный аспект — тестирование ошибок. Например:
```cpp
int divide(int a, int b) {
    if (b == 0) throw std::invalid_argument("zero");
    return a / b;
}

TEST(DivideTest, ThrowsOnZero) {
    EXPECT_THROW(divide(10, 0), std::invalid_argument);
}
```
gtest напрямую поддерживает проверку исключений.

При работе с числами с плавающей точкой нельзя использовать обычное равенство из-за погрешности. Для этого есть специальные макросы:
```cpp
EXPECT_NEAR(0.1 + 0.2, 0.3, 1e-9);
```
Это критично, иначе тесты будут нестабильными.

С точки зрения структуры проекта тесты обычно отделяются от основного кода. Типичная схема:
- `src/` — production код
- `tests/` — unit-тесты

Каждый модуль тестируется отдельно. Это уменьшает связность и упрощает поддержку.
```cpp
// math_ops.h
int add(int a, int b);

// test_math.cpp
#include "math_ops.h"
#include <gtest/gtest.h>

TEST(MathTest, AddWorks) {
    EXPECT_EQ(add(1, 2), 3);
}
```

С практической точки зрения важнее не синтаксис, а выбор тест-кейсов. Хороший unit-тест покрывает:
- нормальные значения
- граничные случаи (0, пустые данные)
- некорректный ввод
- потенциальные ошибки

Пример:
```cpp
TEST(DivideTest, BasicCases) {
    EXPECT_EQ(divide(10, 2), 5);
    EXPECT_EQ(divide(-10, 2), -5);
}

TEST(DivideTest, EdgeCases) {
    EXPECT_THROW(divide(10, 0), std::invalid_argument);
}
```

Отдельно стоит отметить архитектурный момент: тесты должны проверять поведение, а не реализацию. Если тест зависит от внутренней структуры класса, он становится хрупким.

Наконец, важное ограничение: unit-тест не должен работать с файловой системой, сетью и другими внешними ресурсами. Такие зависимости либо изолируются, либо выносятся в отдельные уровни тестирования.
