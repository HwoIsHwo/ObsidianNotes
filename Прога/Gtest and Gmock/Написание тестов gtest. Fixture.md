Если в простых тестах достаточно вызвать функцию и проверить результат, то при тестировании классов почти всегда требуется предварительная инициализация: создание объектов, заполнение данных, настройка зависимостей. Повторять это в каждом тесте — плохая практика. Fixtures позволяют централизовать эту логику.

С технической точки зрения fixture — это класс, наследующийся от `::testing::Test`, внутри которого описывается общее состояние тестов и процедуры инициализации/очистки.

Минимальный пример выглядит так:
```cpp
#include <gtest/gtest.h>

// Тестируемый класс
class Calculator {
public:
    int add(int a, int b) {
        return a + b;
    }
};

// Fixture
class CalculatorTest : public ::testing::Test {
protected:
    Calculator calc;  // общий объект для всех тестов

    void SetUp() override {
        // вызывается перед каждым тестом
        // можно инициализировать состояние
    }

    void TearDown() override {
        // вызывается после каждого теста
        // освобождение ресурсов
    }
};
```

Далее тесты пишутся через `TEST_F`, а не `TEST`. Это принципиально: макрос связывает тест с конкретным fixture и даёт доступ к его полям.

```cpp
TEST_F(CalculatorTest, AddPositive) {
    EXPECT_EQ(calc.add(2, 3), 5);
}

TEST_F(CalculatorTest, AddNegative) {
    EXPECT_EQ(calc.add(-2, -3), -5);
}
```

Каждый тест получает **новый экземпляр fixture**. Жизненный цикл всегда одинаковый — создание объекта, вызов `SetUp()`, выполнение теста, затем `TearDown()` и уничтожение объекта. Это гарантирует изоляцию тестов: изменения состояния в одном тесте не влияют на другие.

Важно правильно понимать область видимости данных. Если переменная объявлена внутри `SetUp()`, она недоступна в тесте — её нужно сделать полем класса:

```cpp
class MyTest : public ::testing::Test {
protected:
    int value;

    void SetUp() override {
        value = 42; // теперь доступно в TEST_F
    }
};
```

Иначе получится типичная ошибка: локальная переменная "теряется" после выхода из `SetUp()`.

Следующий аспект — выбор между конструктором и `SetUp()`. Формально инициализацию можно делать в конструкторе, но это ограничено: внутри конструктора нельзя использовать `ASSERT_*`, потому что он не возвращает `void`. Поэтому в реальных тестах предпочтительно использовать `SetUp()`.

```cpp
void SetUp() override {
    ASSERT_TRUE(initSomething()); // допустимо
}
```

Для тяжёлых операций (например, подключение к БД, загрузка файлов) существует уровень выше — `SetUpTestSuite()` и `TearDownTestSuite()`. Они вызываются один раз на весь набор тестов, а не перед каждым тестом.

```cpp
class HeavyTest : public ::testing::Test {
protected:
    static void SetUpTestSuite() {
        // выполняется один раз
    }

    static void TearDownTestSuite() {
        // выполняется один раз
    }
};
```

Это позволяет избежать лишних затрат времени, но требует аккуратности: такие ресурсы становятся общими, и тесты могут начать зависеть друг от друга.

Практическое применение fixtures наиболее очевидно при тестировании классов с состоянием. Например:

```cpp
class Stack {
public:
    void push(int v) { data.push_back(v); }
    int pop() {
        int v = data.back();
        data.pop_back();
        return v;
    }
    size_t size() const { return data.size(); }

private:
    std::vector<int> data;
};

class StackTest : public ::testing::Test {
protected:
    Stack stack;

    void SetUp() override {
        stack.push(1);
        stack.push(2);
    }
};

TEST_F(StackTest, SizeAfterInit) {
    EXPECT_EQ(stack.size(), 2);
}

TEST_F(StackTest, PopWorks) {
    EXPECT_EQ(stack.pop(), 2);
    EXPECT_EQ(stack.size(), 1);
}
```

Здесь инициализация стека выполняется один раз в `SetUp()`, и все тесты получают одинаковую стартовую конфигурацию.

Концептуально fixtures — это не просто удобство, а способ поддерживать корректную архитектуру тестов. Они:
- устраняют дублирование
- обеспечивают изоляцию
- делают тесты предсказуемыми

При этом есть типичная ошибка — перегруженные fixtures. Если в `SetUp()` слишком много логики, тесты становятся медленными и менее прозрачными. Рекомендуется держать fixture минимальным и выносить специфическую логику в сами тесты.
