Прототип задач в рамках FreeRTOS
![[{7144071C-803D-425F-A42C-EB30FE264650}.png]]
По прерыванию системного таймера (согласно такту ОС) планировщик включается в процесс "выбора" задачи, на выполнение которой переключится МК.
Для каждой задачи выделяется локальный стек (аналогично функциям, чем по сути являются задачи). Стеку позволяет сохранять локальные переменные задачи во время того, как управление передано другой задаче. Также создается небольшой участок памяти - TCB (task control block), в котором находится название задачи, ссылка на функцию задачи, данные о приоритете, о времени создания и о ее статусе.
Цикл жизни задачи:
![[{5A4B59BF-E449-4AA3-BCEB-9EE97773F6E4}.png]]
Точка входа (черная точка) - момент создания задачи в ОС. 

Приоритет задач позволяет выбирать одну задачу из многих для последующего выполнения. То есть планировщик на основе приоритета выбирают самую важную задачу на данный момент времени и передает ей управление. Неактивная задача (idle) имеет нулевой приоритет. При этом планировщик гарантирует, что каждой задаче в процессе выполнения или готовой к выполнению будет выделено процессорное время. Задачи могут иметь одинаковый приоритет. Планировщик гарантирует, что из всех готовых задач для выполнения будет выбрана задача с наивысшим приоритетом.
![[{6DEFCA04-3EFC-4C96-A31A-4AA1A7E44CAA}.png]]

Создание задачи:
![[{B0A6E3EC-BB7E-45A4-88F4-3A69A1173625}.png]]
Аргументы:
1. указатель на функцию задачи
2. имя задачи. Почти не используется планировщиком, только для отладки
3. размер стека. Размер в словах, а не байтах.
4. параметр для функции задачи
5. приоритет задачи. Может быть изменен
6. дескриптор создаваемой задачи. Может использоваться в качестве ссылки на задачу в вызовах других функций API FreeRTOS
Возвращаемых значения два - задача создана и задача не была создана. Основная возможная проблема создания задачи - нехватка памяти в куче для создания задачи, контрольного блока и стека.

В примере выше задача будет расходовать все выделенное ей процессорное время. Но для реальных задач это может быть излишним. То есть необходимо, чтобы задача выполнялась с какой-то периодичностью, а не во все моменты, когда ей передано управление. Тогда можно ввести временную блокировку задачи:
![[{3E6F7CDC-BF37-4748-BD79-A9F541D3DEA3}.png]]
первый метод блокирует задачу на заданное время. Второй - блокирует задачу на заданное количество тактов начиная с определенного такта.
![[{E739A94B-1E8A-4BB7-96E0-9446078E2AE4}.png]]
Таким образом задача расходует столько времени, сколько ей реально нужно (в рамках одного акта передачи управления).

Планировщику для успешной работы всегда нужна задача, которая может получить доступ к ядру МК. Для этого используется задача простоя, занимающая процессорное время, когда нет других задач на выполнение:
![[{F1AC3A06-9FA5-4FE6-A677-CFDC21BD1ADB}.png]]
Пример выше, за счет счетчика в задаче простоя, позволяет приблизительно определить как много тактов процессор простаивает в единицу времени.

Работой планировщика (алгоритмом планирования, выбора задачи для выполнения) можно управлять. 
![[{7A7688F7-DFA7-427A-8D27-B66DA9746E3F}.png]]
По умолчанию режим планирования - упреждающее планирование с фиксированным приоритетом и разделением по времени.
- Фиксированный приоритет - алгоритм планирования не изменяет приоритет задачам. Но не запрещает изменять.
- упреждающий - Если задача с более высоким приоритет, чем выполняемая, становится готовой, то следующей будет выполнена она.
- разделение по времени - имеющееся время разбивается для задач с одинаковым приоритетом.
Если отключить разделение времени, задача будет прекращать свою работу сама, планировщик не будет ее останавливать для выполнения задачи с таким же приоритетом. Но переключает на задачу с более высоким приоритетом.
