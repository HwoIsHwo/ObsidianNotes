Контекст: работа происходит в одной из веток удаленного репозитория (допустим, develop). Нужно для своей части работы создавать новую ветку на основе этой, вносить изменения, и делать пул-реквест обратно в develop.

---
Настройка локального репозитория:
```bash
# Клонирование репозитория
git clone <repo_url>

cd <repo_name>

# Проверка веток
git branch -a

# Переключение на develop (если не по умолчанию)
git checkout develop

# Подтянуть актуальное состояние
git pull origin develop
```
Смысл: cинхронизация с удалённой веткой и работа от актуального состояния.

---
Для каждой задачи
Создание рабочей ветки:
```bash
# Убедиться, что develop актуален
git checkout develop
git pull origin develop

# Создание новой ветки от develop
git checkout -b feature/<task_name>
```
Здесь важно: ветка всегда создаётся от свежего `develop`, иначе появятся лишние конфликты.

Работа с изменениями:
```bash
# Посмотреть статус
git status

# Добавить файлы в индекс
git add <file1> <file2>
# или все изменения
git add .

# Создать коммит
git commit -m "Краткое описание изменений"
```

Перед пушем желательно синхронизироваться с `develop`, если работа заняла время:
```bash
git checkout develop
git pull origin develop

git checkout feature/<task_name>

# Подтянуть изменения из develop в свою ветку
git merge develop
```

Публикация ветки:
```bash
git push -u origin feature/<task_name>
```
Ключ `-u` привязывает локальную ветку к удалённой, дальше можно просто `git push`.

После этого создаётся Pull Request (через GitHub/GitLab UI):
- base: `develop`
- compare: `feature/<task_name>`

Если после ревью нужно внести правки:
```bash
# Вносим изменения
git add .
git commit -m "Fix after review"

# Отправляем в ту же ветку
git push
```

После мержа ветки в `develop`:
```bash
# Обновить develop
git checkout develop
git pull origin develop

# Удалить локальную ветку
git branch -d feature/<task_name>

# Удалить удалённую ветку
git push origin --delete feature/<task_name>
```
