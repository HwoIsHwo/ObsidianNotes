Работа по SSH позволяет аутентифицироваться без ввода логина и пароля при каждом обращении к удалённому репозиторию. Вместо этого используется пара ключей: приватный (хранится локально) и публичный (добавляется в аккаунт Git-сервиса).


---
## Генерация SSH-ключа
Перед созданием ключа проверяется его наличие:
```bash
ls -al ~/.ssh
```
Если файлов вида `id_ed25519` или `id_rsa` нет, создаётся новый ключ. 
```shell
mkdir -p ~/.ssh
chmod 700 ~/.ssh
cd ~/.ssh
```
Рекомендуется использовать алгоритм Ed25519:
```bash
ssh-keygen -t ed25519 -C "your_email@example.com"
ssh-keygen -t rsa -b 4096 -C "ваш_email@example.com"
```
Параметр `-C` — это комментарий, обычно email. После запуска будет предложено:
- указать путь для сохранения (по умолчанию `~/.ssh/id_ed25519`);
- задать парольную фразу (опционально, но рекомендуется).

В результате создаются два файла:
- `id_ed25519` — приватный ключ (никогда не передаётся);
- `id_ed25519.pub` — публичный ключ.

---
## Запуск ssh-agent и добавление ключа
Для управления ключами используется ssh-agent:
```bash
eval "$(ssh-agent -s)"
```
Добавление ключа в агент:
```bash
ssh-add ~/.ssh/id_ed25519
```
Проверка:
```bash
ssh-add -l
```


---
## Добавление публичного ключа в Git-сервис
Содержимое публичного ключа:
```bash
cat ~/.ssh/id_ed25519.pub
```
Ключ копируется целиком и добавляется в настройках аккаунта:
Для GitHub:
- Settings → SSH and GPG keys → New SSH key
Для GitLab:
- Preferences → SSH Keys
Для Bitbucket:
- Personal settings → SSH keys

---
## Проверка соединения
После добавления ключа проверяется доступ:
```bash
ssh -T git@github.com
```
Ожидаемый результат — сообщение об успешной аутентификации.
Если используется другой сервис, меняется домен:
- GitLab: `git@gitlab.com`
- Bitbucket: `git@bitbucket.org`
Если вы видите сообщение, SSH‑подключение к GitHub настроено корректно — можно использовать SSH‑URL репозиториев вида `git@github.com:username/repository.git` для `git clone`, `git pull` и `git push`.

---
## Настройка удалённого репозитория
Если репозиторий уже был клонирован по HTTPS, URL можно заменить:
```bash
git remote -v
git remote set-url origin git@github.com:<user>/<repo>.git
```

Для нового клонирования сразу используется SSH:
```bash
git clone git@github.com:<user>/<repo>.git
```

Типичные проблемы
1. Permission denied (publickey)  
    Причины:
	- ключ не добавлен в ssh-agent;
	- публичный ключ не добавлен в аккаунт;
	- используется неправильный путь к ключу.

2. Несколько ключей  
    Если используется несколько ключей, настраивается файл:
```bash
~/.ssh/config
```
Пример:
```bash
Host github.com
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_ed25519
```

3. Неправильные права доступа
```bash
chmod 700 ~/.ssh
chmod 600 ~/.ssh/id_ed25519
chmod 644 ~/.ssh/id_ed25519.pub
```
