Unit тестирование — это метод проверки программного обеспечения на уровне отдельных компонентов: функций, классов или небольших модулей. Под «unit» понимается минимальная логическая единица, поведение которой можно проверить изолированно.

Основная цель — гарантировать корректность поведения каждого компонента независимо от остальной системы. Это достигается за счёт изоляции: тестируемый код не должен зависеть от внешних факторов, таких как сеть, файловая система или база данных.

С практической точки зрения unit-тест — это обычная функция, которая:
1. подготавливает входные данные
2. вызывает тестируемый код
3. проверяет результат

В C++ это обычно выглядит так:
```cpp
#include <gtest/gtest.h>

int multiply(int a, int b) {
    return a * b;
}

TEST(MultiplyTest, BasicCases) {
    EXPECT_EQ(multiply(2, 3), 6);
    EXPECT_EQ(multiply(-2, 3), -6);
}
```
Ключевой момент — наличие ожидаемого результата. Без него тест не имеет смысла: он не проверяет корректность, а лишь выполняет код.

Изоляция является фундаментальным требованием. Если функция зависит от внешнего ресурса, его необходимо подменить. Для этого используются mock-объекты.
```cpp
class IDataSource {
public:
    virtual ~IDataSource() = default;
    virtual int get() = 0;
};

class Processor {
public:
    Processor(IDataSource* src) : src_(src) {}

    int process() {
        return src_->get() * 2;
    }

private:
    IDataSource* src_;
};
```

В тесте:
```cpp
#include <gmock/gmock.h>

class MockDataSource : public IDataSource {
public:
    MOCK_METHOD(int, get, (), (override));
};

TEST(ProcessorTest, UsesDataSource) {
    MockDataSource mock;

    EXPECT_CALL(mock, get())
        .WillOnce(::testing::Return(5));

    Processor p(&mock);

    EXPECT_EQ(p.process(), 10);
}
```
Здесь тест проверяет только логику `Processor`, а не поведение источника данных.

Unit тестирование отличается от других уровней тестирования. Интеграционные тесты проверяют взаимодействие между компонентами, системные — всю систему целиком. Unit-тесты находятся на нижнем уровне и должны быть максимально быстрыми и независимыми.

Существует несколько базовых принципов, которые определяют качество unit-тестов.

- Первый — детерминированность. Тест всегда должен давать один и тот же результат при одинаковых входных данных. Использование случайных значений, времени или внешних ресурсов нарушает это требование.
- Второй — изоляция. Тест не должен зависеть от состояния других тестов или глобального окружения. Каждый тест должен быть самодостаточным.
- Третий — атомарность. Один тест проверяет одно поведение. Это упрощает диагностику: при падении сразу понятно, что именно сломалось.
- Четвёртый — воспроизводимость. Тест должен одинаково работать на любой машине и в любой среде.
- Пятый — скорость. Unit-тесты должны выполняться быстро, иначе их не будут запускать регулярно.

С точки зрения архитектуры unit-тестирование напрямую влияет на дизайн кода. Чтобы код был тестируемым, он должен:
* иметь явные зависимости (dependency injection)
* использовать интерфейсы
* избегать жёсткой связности

Плохой пример:
```cpp
class Service {
public:
    int run() {
        Database db;
        return db.get();
    }
};
```
Такой код невозможно изолированно протестировать.

Хороший пример:
```cpp
class Service {
public:
    Service(IDatabase* db) : db_(db) {}

    int run() {
        return db_->get();
    }

private:
    IDatabase* db_;
};
```
Теперь зависимость можно подменить.

Отдельный аспект — выбор тест-кейсов. Хороший unit-тест покрывает:
* нормальные сценарии
* граничные значения
* некорректный ввод
```cpp
TEST(DivideTest, HandlesCases) {
    EXPECT_EQ(divide(10, 2), 5);           // нормальный случай
    EXPECT_THROW(divide(10, 0), std::invalid_argument); // ошибка
}
```
При этом важно не стремиться к 100% покрытию любой ценой. Покрытие — это метрика, а не цель. Тесты должны быть осмысленными, а не формальными.

Также существует подход TDD (Test-Driven Development), при котором тесты пишутся до реализации:
1. написать тест
2. убедиться, что он падает
3. реализовать код
4. добиться прохождения теста
Это дисциплинирует разработку и уменьшает количество дефектов.

Наконец, важное ограничение: unit-тестирование не заменяет другие виды тестирования. Оно не выявляет проблемы интеграции, производительности или работы в реальной среде. Это инструмент для проверки локальной логики.
