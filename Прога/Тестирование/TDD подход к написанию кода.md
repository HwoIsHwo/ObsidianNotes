TDD (Test-Driven Development) — это подход к разработке, при котором тесты пишутся до реализации функциональности. Основная идея — формализовать требования к коду в виде тестов и затем реализовывать код так, чтобы эти тесты проходили.

Процесс TDD представляет собой короткий цикл, состоящий из трёх этапов: **Red → Green → Refactor**.

На этапе _Red_ пишется тест, который описывает требуемое поведение, и ожидаемо падает, так как реализация ещё отсутствует.
```cpp
#include <gtest/gtest.h>

int sum(int a, int b); // ещё не реализована

TEST(SumTest, HandlesPositive) {
    EXPECT_EQ(sum(2, 3), 5);
}
```
Компиляция может пройти (если есть объявление), но выполнение теста приведёт к ошибке или undefined behavior. Это и есть индикатор того, что тест действительно проверяет отсутствие реализации.

На этапе _Green_ добавляется минимально необходимый код, чтобы тест прошёл. Важно: не писать «правильную» реализацию сразу, а только ту, которая удовлетворяет текущему тесту.
```cpp
int sum(int a, int b) {
    return 5; // минимальная реализация под текущий тест
}
```
Тест проходит, но реализация очевидно некорректна. Это допустимо на данном этапе, так как цель — двигаться маленькими шагами.

Далее добавляется новый тест, который расширяет требования:
```cpp
TEST(SumTest, HandlesDifferentValues) {
    EXPECT_EQ(sum(1, 1), 2);
}
```
Теперь тест падает, и реализацию приходится обобщить:

```cpp
int sum(int a, int b) {
    return a + b;
}
```
Оба теста проходят. Это завершение цикла _Green_.

На этапе _Refactor_ код улучшается без изменения поведения. Это может включать:
- упрощение логики
- переименование
- устранение дублирования

Важно, что после каждого изменения запускаются тесты — они гарантируют, что поведение не нарушено.

С технической точки зрения TDD заставляет разработчика сначала формализовать контракт функции, а уже затем писать реализацию. Это меняет направление мышления: не «как реализовать», а «что должно получиться».

Ключевой эффект — декомпозиция задачи. Разработка ведётся маленькими шагами, каждый из которых проверяется тестом. Это снижает риск ошибок и упрощает отладку.

При работе с зависимостями TDD тесно связан с использованием mock-объектов. Если компонент зависит от внешней системы, сначала описывается ожидаемое взаимодействие, а затем реализуется логика.
```cpp
class IDataBase {
public:
    virtual int get() = 0;
};

class MockDB : public IDataBase {
public:
    MOCK_METHOD(int, get, (), (override));
};

TEST(ServiceTest, UsesDatabase) {
    MockDB mock;

    EXPECT_CALL(mock, get())
        .WillOnce(::testing::Return(10));

    Service s(&mock);
    EXPECT_EQ(s.process(), 20);
}
```
Здесь тест задаёт контракт: сервис должен вызвать `get()` и использовать результат определённым образом. Реализация строится под этот контракт.

Существует несколько практических правил, которые обеспечивают эффективность TDD.
- шаги должны быть минимальными. Один тест — одно небольшое изменение. Если писать большие тесты сразу, теряется основной эффект метода.
- тест должен быть максимально простым. Сложные тесты труднее поддерживать и анализировать.
- нельзя пропускать фазу рефакторинга. Без неё код постепенно деградирует, даже если тесты проходят.
- важно избегать избыточного обобщения. Код должен расширяться только тогда, когда появляется новый тест, требующий этого.

Типичные ошибки при использовании TDD связаны с нарушением цикла:
- написание реализации до теста. В этом случае TDD превращается в обычное тестирование после факта.
- написание слишком сложных тестов. Это приводит к тому, что приходится реализовывать много логики сразу.
- игнорирование рефакторинга. Код становится громоздким, несмотря на наличие тестов.
- тестирование реализации вместо поведения. Тесты начинают зависеть от внутренней структуры кода и ломаются при любом изменении.

С точки зрения архитектуры TDD приводит к следующим эффектам:
- код становится более модульным
- зависимости становятся явными
- упрощается внедрение интерфейсов и DI
- повышается тестируемость

Однако TDD имеет ограничения. Он плохо подходит для:
- UI
- сложных интеграционных сценариев
- прототипирования

Также он увеличивает начальные затраты времени, особенно на этапе освоения.
