Mock-объекты применяются в ситуациях, когда тестируемый код зависит от внешних компонентов: базы данных, сети, файловой системы или других модулей. Прямая работа с такими зависимостями делает тесты медленными и нестабильными. gmock решает эту проблему за счёт подмены реальных объектов управляемыми реализациями, поведение которых задаётся внутри теста.

Ключевая идея — mock реализует тот же интерфейс, что и реальный объект, но дополнительно позволяет:
- задать ожидаемые вызовы
- контролировать возвращаемые значения
- проверять взаимодействие с зависимостью

Это принципиально отличается от обычного unit-теста: проверяется не только результат, но и сам процесс выполнения.

Рассмотрим типичный сценарий. Есть интерфейс зависимости:
```cpp
class IDataBase {
public:
    virtual ~IDataBase() = default;
    virtual int getValue() = 0;
};
```
И класс, который мы хотим тестировать:
```cpp
class Service {
public:
    Service(IDataBase* db) : db_(db) {}

    int process() {
        return db_->getValue() + 10;
    }

private:
    IDataBase* db_;
};
```
Если использовать реальную БД, тест будет зависеть от внешнего состояния. Вместо этого создаётся mock.
```cpp
#include <gmock/gmock.h>

class MockDataBase : public IDataBase {
public:
    MOCK_METHOD(int, getValue, (), (override));
};
```
Макрос `MOCK_METHOD` генерирует реализацию метода. Важно, что методы должны быть `virtual`, иначе их нельзя переопределить.

Далее задаётся поведение mock-объекта через `EXPECT_CALL`.
```cpp
#include <gtest/gtest.h>

TEST(ServiceTest, UsesDatabaseValue) {
    MockDataBase mock;

    EXPECT_CALL(mock, getValue())
        .Times(1)
        .WillOnce(::testing::Return(5));

    Service service(&mock);
    int result = service.process();

    EXPECT_EQ(result, 15);
}
```
Здесь происходит сразу несколько вещей.

Во-первых, задаётся ожидание: метод `getValue()` должен быть вызван ровно один раз. Во-вторых, определяется возвращаемое значение. В-третьих, после выполнения теста gmock автоматически проверяет, были ли выполнены все ожидания.

Если метод не вызван или вызван с другими параметрами — тест упадёт.

Следующий важный аспект — проверка аргументов. gmock предоставляет механизм matchers.
```cpp
EXPECT_CALL(mock, getValue())
    .Times(::testing::AtLeast(1));
```
Для методов с параметрами:
```cpp
class ICalc {
public:
    virtual int sum(int a, int b) = 0;
};

class MockCalc : public ICalc {
public:
    MOCK_METHOD(int, sum, (int a, int b), (override));
};

TEST(CalcTest, ChecksArguments) {
    MockCalc mock;

    EXPECT_CALL(mock, sum(2, 3))
        .WillOnce(::testing::Return(5));

    EXPECT_EQ(mock.sum(2, 3), 5);
}
```
Если вызвать `sum(1, 3)`, тест не пройдёт — аргументы не совпали.

Отдельно стоит рассмотреть архитектурное требование: чтобы использовать mock, код должен быть спроектирован с возможностью подмены зависимостей. Обычно это достигается через dependency injection.
Плохой вариант:
```cpp
class Service {
public:
    int process() {
        RealDB db; // жёсткая зависимость
        return db.getValue();
    }
};
```
Хороший вариант:
```cpp
class Service {
public:
    Service(IDataBase* db) : db_(db) {}

private:
    IDataBase* db_;
};
```
Без этого mock использовать невозможно, потому что объект создаётся внутри класса.

Также важно понимать разницу между mock и fake. Fake — это упрощённая реализация (например, in-memory БД), а mock — это объект с заранее заданными ожиданиями, который проверяет взаимодействие.

Практический смысл использования gmock сводится к трём задачам:
- изоляция теста от внешней среды
- проверка взаимодействия между компонентами
- возможность тестировать код до реализации зависимостей
