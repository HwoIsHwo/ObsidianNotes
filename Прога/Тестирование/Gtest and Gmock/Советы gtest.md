Практическое использование gtest/gmock быстро выходит за рамки синтаксиса. Основные проблемы начинаются не в написании тестов, а в их поддержке: избыточные проверки, хрупкие mock’и, дублирование и зависимость от реализации. 

Первый базовый принцип — тест должен проверять **одно поведение**. Если в одном тесте проверяется несколько сценариев, он становится трудно диагностируемым. Это соответствует классическому шаблону Arrange–Act–Assert: сначала подготовка, затем выполнение, затем проверка.

```cpp
TEST(ServiceTest, ReturnsCorrectValue) {
    // Arrange
    MockDataBase mock;
    EXPECT_CALL(mock, getValue())
        .WillOnce(::testing::Return(5));

    Service service(&mock);

    // Act
    int result = service.process();

    // Assert
    EXPECT_EQ(result, 15);
}
```

Следующий важный момент — баланс строгости mock-объектов. gmock позволяет очень точно описывать ожидания, но чрезмерная строгость делает тесты нестабильными. Любое изменение реализации (даже без изменения поведения) приводит к падению теста. Рекомендуется задавать только те ожидания, которые действительно важны для поведения системы.

```cpp
// Плохо — слишком жёстко
EXPECT_CALL(mock, getValue()).Times(1);

// Лучше — допускаем гибкость
EXPECT_CALL(mock, getValue()).Times(::testing::AtLeast(1));
```

С этим связан ещё один принцип: не проверять лишнее. Если тест проверяет результат функции, не нужно одновременно проверять все внутренние вызовы, если они не являются частью контракта. В противном случае тест начинает зависеть от реализации, а не от поведения.

Отдельная проблема — дублирование. Часто один и тот же набор `EXPECT_CALL` повторяется в разных тестах. Это сигнал к тому, что тесты плохо структурированы. Решение — вынос общей логики в fixtures или вспомогательные функции (с учётом того, что сами `EXPECT_CALL` нельзя просто обернуть в функцию без последствий для gmock).

```cpp
class ServiceTest : public ::testing::Test {
protected:
    MockDataBase mock;
};
```

Далее — работа с `WillOnce`. Частая ошибка — цепочки из большого количества `WillOnce`, которые делают тест громоздким и трудно читаемым. В таких случаях лучше использовать `WillRepeatedly`, если поведение одинаковое, либо пересмотреть сам тест.

```cpp
EXPECT_CALL(mock, getValue())
    .WillRepeatedly(::testing::Return(5));
```

Следующий важный аспект — выбор типа mock. В gmock есть разные режимы (nice, naggy, strict). Жёсткие mock’и (`StrictMock`) проверяют всё и сразу, но приводят к хрупким тестам. На практике чаще используют более мягкие варианты, чтобы тест фиксировал только существенные ошибки.

Также важно не злоупотреблять mock’ами. Если тест требует сложной системы mock-объектов, это обычно признак плохой архитектуры. В таких случаях стоит пересмотреть дизайн: разбить класс, ввести интерфейсы, упростить зависимости. Mock — это инструмент, а не цель.

```cpp
// Плохой дизайн — жёсткая зависимость
class Service {
public:
    int process() {
        RealDB db;
        return db.getValue();
    }
};

// Хороший дизайн — внедрение зависимости
class Service {
public:
    Service(IDataBase* db) : db_(db) {}
private:
    IDataBase* db_;
};
```

Отдельное правило — осмысленные имена тестов. Название должно описывать поведение и ожидаемый результат. Это особенно важно при большом количестве тестов.

```cpp
TEST(UserServiceTest, CreateUserReturnsErrorWhenNameEmpty) {
    // ...
}
```

Это упрощает анализ падений без просмотра кода.

Ещё один практический момент — безопасность тестов. Потенциально опасные операции (разыменование указателя, доступ к данным) должны предваряться `ASSERT_*`, чтобы избежать краха теста.

```cpp
ASSERT_NE(ptr, nullptr);
EXPECT_EQ(ptr->value, 10);
```

Наконец, стоит понимать границы unit-тестирования. Не нужно пытаться протестировать всё через gtest/gmock. Взаимодействие с файловой системой, сетью и другими подсистемами лучше выносить в интеграционные тесты. Mock’и используются именно для изоляции, а не для полной эмуляции всей системы.
