При unit-тестировании ключевой задачей является изоляция тестируемого кода от внешних зависимостей. Для этого используются различные виды подменных объектов, наиболее распространённые из которых — **mock** и **fake**. Несмотря на схожую цель, они решают разные задачи и применяются в разных ситуациях.

Подмена зависимости требуется, когда реальный компонент:
- медленный (например, база данных)
- нестабильный (сеть)
- недоступен (ещё не реализован)
- усложняет воспроизводимость теста
Вместо него используется контролируемая реализация.

---
## Fake объекты
Fake — это упрощённая, но рабочая реализация зависимости. Она реализует тот же интерфейс, но с более простой логикой, часто in-memory.

Пример:
```cpp
class IDataBase {
public:
    virtual ~IDataBase() = default;
    virtual void save(int v) = 0;
    virtual int load() = 0;
};

// Fake-реализация
class FakeDataBase : public IDataBase {
public:
    void save(int v) override {
        data_ = v;
    }

    int load() override {
        return data_;
    }

private:
    int data_ = 0;
};
```

Такой объект ведёт себя «почти как настоящий», но без внешних зависимостей. Его удобно использовать, когда важен результат, а не процесс взаимодействия.
```cpp
TEST(DatabaseTest, SaveAndLoad) {
    FakeDataBase db;

    db.save(10);
    EXPECT_EQ(db.load(), 10);
}
```

Fake-объекты обладают следующими свойствами:
- имеют реальную (но упрощённую) логику
- не требуют настройки поведения
- обычно не проверяют вызовы методов
Они хорошо подходят для интеграционных тестов на уровне модулей.

---
## Mock объекты
Mock — это объект, который не реализует реальную логику, а задаёт ожидаемое поведение и проверяет взаимодействие.

```cpp
#include <gmock/gmock.h>

class IDataBase {
public:
    virtual ~IDataBase() = default;
    virtual int load() = 0;
};

class MockDataBase : public IDataBase {
public:
    MOCK_METHOD(int, load, (), (override));
};
```
Основное отличие — поведение задаётся в тесте:
```cpp
TEST(ServiceTest, UsesDatabase) {
    MockDataBase mock;

    EXPECT_CALL(mock, load())
        .Times(1)
        .WillOnce(::testing::Return(5));

    Service s(&mock);
    EXPECT_EQ(s.process(), 10);
}
```
Здесь проверяется не только результат, но и факт вызова метода `load()`. Если метод не будет вызван или вызван с неправильными параметрами, тест завершится ошибкой.

Mock-объекты позволяют:
- задать ожидаемые вызовы (`EXPECT_CALL`)
- контролировать возвращаемые значения (`Return`, `Invoke`)
- проверять количество вызовов (`Times`)
Это делает их инструментом для тестирования поведения.

---
## Ключевое различие
Различие между fake и mock можно сформулировать так:
- Fake → проверка состояния (что получилось)
- Mock → проверка поведения (как это было получено)
Fake выполняет реальную работу, mock — только имитирует и контролирует.

## Архитектурные требования
Для использования как fake, так и mock, код должен быть спроектирован с учётом подмены зависимостей. Это достигается через интерфейсы и dependency injection.
```cpp
class Service {
public:
    Service(IDataBase* db) : db_(db) {}

    int process() {
        return db_->load() * 2;
    }

private:
    IDataBase* db_;
};
```
Без этого невозможно заменить реальную зависимость на тестовую.
Жёсткая привязка к реализации делает тестирование практически невозможным:
```cpp
class Service {
public:
    int process() {
        RealDB db; // нельзя подменить
        return db.load();
    }
};
```

---
## Когда использовать
Fake используется, когда:
- нужна рабочая логика без внешних зависимостей
- важен результат, а не процесс
- поведение достаточно простое

Mock используется, когда:
- нужно проверить взаимодействие между компонентами
- важно количество и параметры вызовов
- зависимость сложная или недетерминированная
На практике оба подхода часто комбинируются.

---
## Типичные ошибки
- чрезмерное использование mock. Если тест требует большого количества `EXPECT_CALL`, это признак избыточной связности или плохого дизайна.
- тестирование реализации через mock. Например, проверка всех внутренних вызовов вместо проверки результата. Это делает тесты хрупкими.
- использование fake там, где важно поведение. Fake не позволяет проверить, как именно используется зависимость.
- слишком сложные fake-реализации. В этом случае они превращаются в альтернативную систему, что увеличивает сложность.
