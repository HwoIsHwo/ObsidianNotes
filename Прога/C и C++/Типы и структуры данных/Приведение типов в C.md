В C приведение типов — это либо явный cast, который вы пишете сами, либо неявные преобразования, которые компилятор выполняет по правилам языка. Второе встречается чаще и именно там скрывается большинство ошибок.

Начать нужно с того, что почти любое выражение в C проходит через набор стандартных преобразований. Самый первый этап — integer promotions. Все типы уже `int` (например `char`, `short`, `enum`) перед арифметикой сначала приводятся к `int` или `unsigned int`. Поэтому выражение
```C
char a = 100;
char b = 100;
int c = a + b;   /* a и b сначала становятся int, затем складываются */
```
никогда не выполняется в 8 битах — сложение происходит как минимум в `int`.

Далее идут usual arithmetic conversions. Если операнды разного типа, компилятор приводит их к «общему» типу по строгим правилам ранга и знаковости. Самый опасный случай — смешивание `signed` и `unsigned`. Если оба одного ранга (например `int` и `unsigned int`), то `int` приводится к `unsigned int`. Отрицательное значение при этом превращается в большое положительное по модулю 2^N. Это не UB и не overflow — это корректная, но часто неожиданная семантика.
Например:
```C
unsigned int a = 1;
int b = -2;

if (a + b > 0) {
    /* b сначала превращается в unsigned,
       затем выполняется unsigned-арифметика */
}
```
Здесь `b` становится большим положительным числом, и сравнение почти наверняка окажется истинным.

Отдельная категория — преобразования при присваивании. Когда вы пишете
```C
int x = 3.14;
```
значение `double` неявно приводится к `int` с усечением дробной части. Это допустимо, но теряет информацию. Если значение выходит за диапазон целевого типа, поведение может быть неопределённым (например, при приведении очень большого `double` к `int`).

С указателями правила другие. Любой объектный указатель может быть приведён к `void*` и обратно без потери информации. Это основа универсальных интерфейсов вроде `malloc`. Но приведение между несовместимыми типами указателей — формально допустимо на уровне синтаксиса, однако последующее разыменование может приводить к неопределённому поведению из-за strict aliasing.
```C
int x = 42;
float *p = (float*)&x;  /* компилируется */
float y = *p;           /* UB по strict aliasing */
```
Явное приведение типа — это конструкция `(type)expression`. Оно просто сообщает компилятору: трактуй результат выражения как этот тип. Само по себе приведение не гарантирует корректности. Если вы приводите `double` к `int`, будет усечение. Если приводите большой `int` к `char`, возможна потеря старших битов. Если приводите указатель к несовместимому типу и разыменовываете — возможен UB.

Отдельно стоит понимать разницу между приведением значения и приведением типа выражения. Например:
```C
int a = 5;
int b = 2;

double r = (double)a / b;
```
Здесь важно привести хотя бы один операнд до деления, иначе `a / b` выполнится как целочисленное деление, и только потом результат будет расширен до `double`.

Есть ещё преобразования к `_Bool`: любое ненулевое значение становится `1`, ноль остаётся нулём. В условиях `if`, `while` и в логических операторах также происходит приведение к логическому контексту.

Критически важно помнить: явный cast не «исправляет» неопределённое поведение. Он может подавить предупреждение компилятора, но не меняет фундаментальных правил. Например, снятие `const` через `(int*)` и последующая запись в реально `const` объект остаётся UB, даже если компилятор молчит.

Если обобщить, система преобразований в C устроена так, чтобы минимизировать количество явных аннотаций, но цена за это — сложные правила неявного продвижения типов. Большинство скрытых ошибок в числовой логике связано именно с integer promotions и с переходом от `signed` к `unsigned`. Поэтому в реальном коде полезно избегать смешивания знаковости и внимательно относиться к предупреждениям компилятора.
