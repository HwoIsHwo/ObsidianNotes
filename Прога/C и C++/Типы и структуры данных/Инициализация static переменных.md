Порядок инициализации `static`-объектов зависит от их области и единицы трансляции.
Сначала общее правило: все объекты со **static storage duration** инициализируются до входа в `main()`. Но внутри этого этапа есть два разных шага.
Сначала выполняется **zero-initialization** — вся статическая память обнуляется. Затем выполняется **dynamic initialization** — вызовы конструкторов и вычисление сложных инициализаторов.

Теперь про порядок.
В пределах **одной единицы трансляции** (одного `.cpp`) глобальные и статические объекты инициализируются в порядке их определения в файле. Это гарантировано стандартом.
Между **разными единицами трансляции** порядок не определён. Линкер может расположить их как угодно. Это и есть известная проблема — _static initialization order fiasco_.
```C
// a.cpp
extern int getB();
int A = getB();

// b.cpp
int B = 42;
int getB() { return B; }
```
Если `A` инициализируется раньше `B`, то `getB()` прочитает ещё не инициализированный `B`. Формально это неопределённое поведение.

Что именно произойдёт — зависит от платформы. Может быть ноль (если объект уже прошёл zero-initialization), может быть мусор, может быть корректное значение — но полагаться на это нельзя.

Важно понимать: zero-initialization гарантированно происходит раньше всего. Поэтому неинициализированные глобальные переменные сначала становятся равны нулю. Но если переменная имеет динамический конструктор, её "настоящая" инициализация может ещё не выполниться.

Локальные `static` внутри функции работают иначе. Их инициализация выполняется при первом входе в функцию. Начиная с C++11, это потокобезопасно. Поэтому такой подход часто используют для устранения проблемы порядка инициализации:
```C
int& getB() {
    static int B = 42;
    return B;
}
```
Здесь `B` будет создана при первом вызове `getB()`, и порядок становится контролируемым.

Если зависимая переменная инициализируется раньше своей зависимости, возникает неопределённое поведение. Это не просто “логическая ошибка”, а формально UB.

Чтобы избежать этого, применяют:
- паттерн “construct on first use” (локальный static в функции),
- отказ от глобальных зависимостей,
- явную инициализацию в `main()`.

В целом правило простое: глобальные `static`-объекты в разных `.cpp` не должны зависеть друг от друга напрямую.
