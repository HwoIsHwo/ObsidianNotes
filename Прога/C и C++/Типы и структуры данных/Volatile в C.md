`volatile` — это квалификатор типа, который запрещает компилятору оптимизировать обращения к объекту так, будто его значение не может измениться «само по себе». Он говорит компилятору: каждое чтение и каждая запись должны реально происходить в памяти, в том порядке, в котором они записаны в программе (с учётом правил языка).

Ключевой момент — `volatile` не делает операции атомарными, не добавляет синхронизацию потоков и не решает проблемы гонок. Он влияет только на оптимизации доступа к памяти.

Самый типичный случай — взаимодействие с аппаратурой через memory-mapped регистры. Представим, что регистр статуса устройства расположен по фиксированному адресу и его значение может меняться независимо от программы. Если не использовать `volatile`, компилятор может закэшировать значение в регистре процессора и превратить цикл ожидания в бесконечный.

```C
#define STATUS_REG ((volatile unsigned int*)0x40000000)

void wait_ready(void)
{
    /* Регистр меняется аппаратно, поэтому каждое чтение
       должно реально происходить из памяти */
    while ((*STATUS_REG & 1u) == 0u) {
        /* ожидание */
    }
}
```

Без `volatile` компилятор имел бы право прочитать `*STATUS_REG` один раз и больше к нему не обращаться, потому что в рамках чистой модели языка значение не меняется.

Второй распространённый сценарий — переменные, изменяемые из обработчика сигнала или прерывания. Например:
```C
#include <signal.h>
#include <stdio.h>

volatile sig_atomic_t stop = 0;

void handler(int signo)
{
    stop = 1;  /* может быть вызвано асинхронно */
}

int main(void)
{
    signal(SIGINT, handler);

    while (!stop) {
        /* работа */
    }

    puts("stopped");
    return 0;
}
```
Переменная `stop` изменяется вне основного потока исполнения. Если убрать `volatile`, компилятор может считать, что в теле `while` значение не меняется, и оптимизировать цикл в бесконечный.

В старом коде `volatile` иногда использовали для межпоточного взаимодействия:
```C
volatile int flag = 0;

void worker(void)
{
    flag = 1;
}

void wait_flag(void)
{
    while (flag == 0) {
        /* ожидание */
    }
}
```

Однако это некорректная синхронизация. `volatile` не гарантирует ни атомарности, ни порядка операций между потоками. В C11 для этого существуют атомарные типы из `<stdatomic.h>`, а в C++ — `std::atomic`. Использовать `volatile` как примитив синхронизации — ошибка.

Важно понимать ещё одну деталь: `volatile` относится к типу, а не к объекту «в целом». Если есть указатель `int *p`, и объект объявлен как `volatile int x`, то доступ через `p = &x; *p` формально нарушает квалификацию типа и ведёт к неопределённому поведению. Квалификаторы должны сохраняться.

Также `volatile` не запрещает компилятору оптимизировать сам код, если это не влияет на наблюдаемые обращения к объекту. Например, выражение `v = v;` для `volatile` переменной не может быть удалено, потому что это реальная запись. Но вычисления вокруг неё могут быть перестроены, если стандарт это допускает.

Отдельный случай — комбинация `const volatile`. Это типично для регистров, которые нельзя изменять программно, но которые меняются аппаратно:
```C
#define DEVICE_ID (*(const volatile unsigned int*)0x40000004)
```

Здесь `const` запрещает запись через этот тип, а `volatile` требует реального чтения при каждом обращении.

Если обобщить, `volatile` нужен там, где значение может изменяться вне логики текущего потока исполнения — аппаратурой, сигналом или внешним агентом, — и где важно, чтобы каждое обращение к памяти действительно произошло. Во всех остальных случаях его применение либо избыточно, либо маскирует более серьёзные архитектурные ошибки.
