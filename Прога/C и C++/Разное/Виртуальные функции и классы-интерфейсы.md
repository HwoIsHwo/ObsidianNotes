В C++ нет отдельного ключевого слова `interface`. Под интерфейсом обычно понимают абстрактный класс, который
- содержит только виртуальные функции (чаще всего — чисто виртуальные),
- не имеет состояния,
- имеет виртуальный деструктор.

Типичный пример:
```
struct IPort {
    virtual ~IPort() = default;

    virtual void open() = 0;
    virtual void close() = 0;
    virtual std::size_t write(const void* data, std::size_t size) = 0;
};
```
`= 0` означает, что функция **чисто виртуальная**. Класс становится абстрактным, и объект такого типа создать нельзя.

Создаётся только производный класс:
```
class SerialPort : public IPort {
public:
    void open() override { /* ... */ }
    void close() override { /* ... */ }
    std::size_t write(const void*, std::size_t) override { return 0; }
};
```


Виртуальность обеспечивает **динамический полиморфизм** — выбор реализации функции во время выполнения, а не компиляции.
```
void use_port(IPort& port) {
    port.open();
}
```
Если передать `SerialPort`, будет вызван `SerialPort::open()`, даже если параметр имеет тип `IPort&`.
Без `virtual` вызов определялся бы **по статическому типу** (`IPort`), а не по реальному типу объекта.

Когда в классе появляется хотя бы одна виртуальная функция, компилятор:
1. Создаёт **таблицу виртуальных функций** (vtable).
2. Добавляет в объект скрытый указатель на эту таблицу — обычно `vptr`.

Схематично объект выглядит так:
\[ vptr ] \[ поля класса ]
А vtable — это массив указателей на функции:
vtable for SerialPort:
+---------------------------+
| &SerialPort::open         |
| &SerialPort::close        |
| &SerialPort::write        |
| &SerialPort::~SerialPort  |
+---------------------------+
Когда выполняется:
```
IPort* p = new SerialPort;
p->open();
```
Происходит:
1. Из объекта берётся `vptr`.
2. По фиксированному смещению в vtable выбирается нужная функция.
3. Выполняется вызов через указатель.
Это и есть динамическая диспетчеризация.

Если производный класс переопределяет виртуальную функцию, в его vtable будет записан новый адрес функции.
Если не переопределяет — остаётся адрес базовой реализации.
Пример:
```
struct Base {
    virtual void f() { std::cout << "Base\n"; }
};

struct Derived : Base {
    void f() override { std::cout << "Derived\n"; }
};
```

Если класс используется полиморфно:
```
IPort* p = new SerialPort;
delete p;
```
Удаление через базовый указатель должно вызвать:
```
~SerialPort()
~IPort()
```
Если деструктор не виртуальный — вызовется только `~IPort()`. Это приводит к UB.

Поэтому правило простое:  **если есть хотя бы одна виртуальная функция — деструктор почти всегда должен быть виртуальным.**

Во время выполнения конструктора базового класса динамический тип объекта ещё не считается производным. Поэтому виртуальные вызовы ведут себя как невиртуальные.
