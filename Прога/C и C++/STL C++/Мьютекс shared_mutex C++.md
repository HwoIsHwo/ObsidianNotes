`std::shared_mutex` — это примитив синхронизации из C++17, позволяющий разделять доступ на режим чтения и записи. В отличие от обычного `std::mutex`, он поддерживает два типа блокировок: совместную (shared) и эксклюзивную (unique). Идея в том, что несколько потоков могут одновременно читать одни и те же данные, если никто их не изменяет.

Обычный `std::mutex` работает проще. Он представляет собой бинарную блокировку: либо свободен, либо захвачен. Если один поток вошёл в критическую секцию, все остальные — и читатели, и писатели — ждут. Это универсально и предсказуемо, но плохо масштабируется при read-heavy нагрузке.

Пример с `std::mutex`:
```
std::mutex m;

void read() {
    std::lock_guard<std::mutex> lock(m);
}

void write() {
    std::lock_guard<std::mutex> lock(m);
}
```

Здесь чтение и запись полностью сериализованы.
`std::shared_mutex` вводит две разные модели блокировки:
- `std::shared_lock` — для чтения
- `std::unique_lock` — для записи
Пример:
```
#include <shared_mutex>

std::shared_mutex m;

void read() {
    std::shared_lock lock(m);
}

void write() {
    std::unique_lock lock(m);
}
```
Теперь несколько потоков могут одновременно находиться в `read()`. Но как только появляется `write()`, он должен дождаться освобождения всех shared-блокировок. И наоборот — пока активна запись, чтения блокируются.

Важно понимать стоимость. `shared_mutex` сложнее реализован внутри ОС или runtime. Он хранит счётчик читателей и состояние писателя, поэтому операции блокировки обычно дороже, чем у `mutex`. Поэтому использовать его имеет смысл только тогда, когда чтений действительно много и они могут выполняться параллельно.
Есть ещё один важный аспект — starvation. В некоторых реализациях возможно «голодание писателя», когда постоянный поток читателей не даёт записи получить блокировку. Стандарт это поведение не нормирует строго.

Теперь различие на уровне API:
`std::mutex`
- `lock()`
- `unlock()`
- `std::lock_guard`
- `std::unique_lock`

`std::shared_mutex`
- `lock()` / `unlock()` — эксклюзивный режим
- `lock_shared()` / `unlock_shared()` — режим чтения
- `std::shared_lock`
- `std::unique_lock`
То есть `shared_mutex` можно использовать и как обычный `mutex`, но наоборот — нельзя.

`std::mutex` даёт только эксклюзивную блокировку. В каждый момент времени внутрь критической секции может зайти только один поток — независимо от того, читает он данные или изменяет. Если чтений много, а записей мало, это создаёт лишнюю сериализацию. Потоки будут стоять в очереди даже тогда, когда они просто читают неизменяемые данные.

`std::shared_mutex` поддерживает два режима:
shared-lock — несколько потоков могут одновременно читать,  
unique-lock — только один поток может писать.
