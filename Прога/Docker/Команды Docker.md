Краткое описание опций и команд можно посмотреть командой "docker" в терминале. Также можно посмотреть полную информацию о команде, прописав в терминале: "docker COMMAND --help".

Команды:
<u>run</u> - запуск контейнера. Если контейнера нет, то может выполнить докачку.
<u>pull</u> - скачивание образа
<u>info</u> - получение полной информации о docker
<u>images</u> - информация о всех образах внутри docker.
<u>ps</u> - весь список доступных контейнеров
<u>stop</u> - останавливает один или несколько запущенных контейнеров.
<u>start</u> - запуск контейнера
<u>pause</u> - поставить контейнер на паузу.
<u>unpause</u> - продолжить выполнение контейнера
<u>restart</u> - Перезапуск контейнера.
<u>login</u> - авторизация в docker
<u>logout</u> - выход из учетки.

- название образа для использования

<u>Образ (image) </u>— это неизменяемый шаблон. По сути, это файловая система плюс метаданные: зависимости, бинарники, переменные окружения, команда запуска. Он собирается, например, через Dockerfile и хранится в виде слоёв. Образ сам по себе не выполняется.

<u>Контейнер (container) </u>— это уже запущенный экземпляр образа. Когда ты создаёшь контейнер, Docker берет образ, добавляет поверх него тонкий слой записи (read-write слой) и запускает процесс с изолированными ресурсами (namespace, cgroups).
Если провести аналогию: образ — это бинарник программы, контейнер — это запущенный процесс этой программы с конкретным состоянием.

<u>Docker Engine</u> — среда выполнения, состоящая из демона (dockerd), API и CLI. Демон управляет образами, контейнерами, сетями и томами, а пользователь взаимодействует с ним через клиент.

<u>Dockerfile</u> — текстовое описание сборки образа. Каждая инструкция (FROM, RUN, COPY и т.д.) создаёт слой. Важно учитывать кэширование: изменение одной инструкции инвалидирует все последующие слои.

<u>Layer</u> (слой) — неизменяемый фрагмент файловой системы образа. Образы состоят из стека слоёв. Повторное использование слоёв снижает объём хранения и ускоряет сборку.

<u>Union filesystem</u> — механизм объединения слоёв в единую файловую систему. Контейнер видит «собранное» дерево, хотя физически оно состоит из нескольких read-only слоёв и одного read-write слоя.

<u>Container runtime</u> — компонент, который непосредственно запускает контейнеры (например, runc). Он использует возможности ядра Linux (namespaces, cgroups) для изоляции.

<u>Namespace</u> — механизм изоляции ресурсов (PID, сеть, файловая система и т.д.) между контейнерами. Каждый контейнер получает своё логическое пространство.

<u>Cgroups</u> — механизм ограничения и учёта ресурсов (CPU, память, I/O). Позволяет задавать лимиты контейнеру.

<u>Registry</u> — хранилище образов (например, Docker Hub). Поддерживает версии (теги) и распространение образов.

<u>Tag</u> — метка версии образа (например, ubuntu:22.04). Один образ может иметь несколько тегов, указывающих на разные версии или конфигурации.

<u>Repository</u> — логическая группа образов с разными тегами. Обычно соответствует одному приложению.

<u>Volume</u> — управляемое Docker-хранилище для данных вне контейнера. Используется для персистентности.

<u>Bind mount</u> — прямое монтирование директории хоста в контейнер. В отличие от volume, полностью зависит от структуры хоста.

<u>Network</u> — виртуальная сеть Docker. Обеспечивает связь контейнеров. Основные режимы: bridge (по умолчанию), host, overlay.

<u>Port mapping </u>— проброс порта контейнера на порт хоста (например, 8080:80). Делает сервис доступным извне.

<u>Entrypoint</u> — основной исполняемый файл контейнера. Обычно фиксирует «что запускать».

<u>CMD </u>— аргументы по умолчанию для entrypoint или самостоятельная команда, если entrypoint не задан.

<u>Environment variables</u> — переменные окружения, передаваемые контейнеру. Используются для конфигурации без пересборки образа.

<u>Build context</u> — набор файлов, доступных при сборке образа. Передаётся демону при выполнении docker build.

<u>.dockerignore</u> — файл исключений для build context. Позволяет уменьшить объём передаваемых данных.

<u>Image cache</u> — механизм повторного использования слоёв при сборке. Критичен для оптимизации времени сборки.

<u>Container lifecycle</u> — состояния контейнера: created, running, paused, stopped, removed. Управляется командами start/stop/restart.

<u>Logs </u>— стандартный вывод контейнера. Docker агрегирует stdout/stderr процесса.

<u>Exec</u> — запуск дополнительной команды внутри уже работающего контейнера.

<u>Compose</u> — инструмент для описания многоконтейнерных приложений в YAML (Docker Compose). Позволяет поднимать стек сервисов одной командой.

<u>Overlay network </u>— распределённая сеть для связи контейнеров на разных хостах. Используется в кластерах.

<u>Swarm</u> — встроенный оркестратор Docker для управления кластером узлов и сервисов.

<u>Service (в Swarm)</u> — декларация желаемого состояния (сколько экземпляров контейнера должно работать, с какими параметрами).
